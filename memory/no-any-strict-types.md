## 强制要求：类型系统严格性

**完善的类型系统是本项目的核心机制，绝对不允许出现 `any` 类型！**

### 执行规则

1. **禁止使用 `any`** — 无论是显式标注还是隐式推断
2. **遇到类型问题必须询问用户** — 不要私自处理或隐藏类型问题
3. **必须提供具体类型** — 使用 `unknown`、具体接口、泛型等替代方案
4. **类型错误不能忽略** — 必须正视并解决，不能用 `any` 绕过

### 常见场景处理

- **不确定类型时**：使用 `unknown` 而非 `any`
- **需要类型断言时**：先询问用户确认类型
- **第三方库类型缺失时**：询问用户如何处理（声明文件 or 类型断言）
- **复杂类型推导失败时**：不要用 `any` 简化，而是询问正确的类型定义

### 为什么如此重要

- 类型系统是项目的**核心机制**
- `any` 会破坏类型安全，导致运行时错误难以追踪
- 低代码平台需要严格的类型保证来确保 Schema 驱动的正确性
- 类型问题应该在编译期暴露，而非运行时

**How to apply:** 在任何代码修改中，如果遇到类型问题或需要使用 `any`，必须暂停并询问用户如何处理。绝不自行决定使用 `any` 或其他类型捷径。
